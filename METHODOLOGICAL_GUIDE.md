# Методическое пособие: Большие вызовы: T-банк

Это пособие помогает провести этап погружения команды в проект. Рабочие
инструкции для команды находятся в [README.md](README.md). Здесь описана логика
этапа, цели блоков и ожидаемый результат.

## Цель этапа

Познакомить команду с тремя вещами:

1. Что такое SymPy как пользовательская библиотека.
2. Как выглядит баг в поведении библиотеки на ручном примере.
3. Как команда переходит от ручного воспроизведения к тестам в bug-ветке.

После этапа команда должна уметь:

- развернуть локальное окружение;
- переключаться между `sirius-light-buggy` и `sirius-light-golden`;
- запускать примеры использования SymPy;
- запускать все учебные тесты и один конкретный тест;
- читать ошибку теста как описание ожидаемого поведения.

## Структура fork

В репозитории есть две учебные ветки:

- `sirius-light-golden` - эталонная ветка с исправленным поведением.
- `sirius-light-buggy` - рабочая ветка с 10 внесенными регрессиями.

Обе ветки используют один и тот же тестовый файл:

```bash
sirius_tests/test_light_bugs.py
```

Полный test suite SymPy на этом этапе не нужен. Команда работает только с
небольшим набором учебных тестов.

## Блок 1. Продуктовое погружение

Задача блока - показать, что баг в библиотеке виден не только в тестах, но и в
обычном пользовательском сценарии.

Рекомендуемый ход:

1. Открыть README и пройти раздел "Что такое SymPy".
2. Запустить пример с производной.
3. Запустить пример с матрицей.
4. Перейти к примеру с `mathematica_code(Max/Min)`.
5. Сначала запустить пример в `sirius-light-buggy`.
6. Затем запустить тот же пример в `sirius-light-golden`.
7. Зафиксировать разницу: в buggy-ветке пользователь получает ошибку, в
   golden-ветке получает корректный результат.

Второй пример с `Point` нужен для закрепления: команда видит другой тип ошибки,
но тот же принцип сравнения buggy/golden.

## Блок 2. Работа с bug-веткой

Задача блока - перевести команду от ручного примера к инженерному циклу:
воспроизвести тест, найти код, исправить, повторить проверку.

Рекомендуемый ход:

1. Переключиться на `sirius-light-buggy`.
2. Выполнить `./scripts/setup_sirius.sh`.
3. Запустить `./scripts/run_sirius_tests.sh`.
4. Показать, что падение тестов - ожидаемое стартовое состояние.
5. Запустить один конкретный тест, например:

```bash
../.venv/bin/python -m pytest -q sirius_tests/test_light_bugs.py::test_Function
```

6. Разобрать ошибку как формулировку ожидаемого поведения.
7. Найти связанный участок кода в папке `sympy/`.
8. После исправления запускать сначала конкретный тест, затем весь
   `sirius_tests/test_light_bugs.py`.

## Ожидаемый результат этапа погружения

К концу этапа погружения команда не обязана исправить все баги. Достаточно,
если команда:

- понимает различие между `buggy` и `golden`;
- умеет запускать окружение и тесты;
- умеет вручную воспроизвести 1-2 бага;
- понимает, что тест фиксирует ожидаемое поведение библиотеки;
- может назвать файл теста и примерный файл кода, который относится к багу.

## Что не нужно делать в этом блоке

- Не запускать полный test suite SymPy.
- Не начинать с архитектуры всей библиотеки.
- Не требовать сразу исправить все 10 багов.
- Не смешивать ручное воспроизведение и тесты: сначала поведение как пользователь,
  затем проверка как инженер.

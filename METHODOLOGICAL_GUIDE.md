# Методическое пособие: Большие вызовы: T-банк

Это пособие помогает провести этап погружения команды в проект. Рабочие
инструкции для команды находятся в [README.md](README.md). Здесь описана логика
этапа, цели блоков и ожидаемый результат.

## Структура репозитория

Этот репозиторий - копия репозитория SymPy, в которую добавлены учебные ветки,
тесты и описания багов для командной работы.

Основные части репозитория:

- `sympy/` - код библиотеки SymPy.
- `bugs_cards/` - карточки багов, перечисленных для работы команды
  в ветке `sirius-light-buggy`.
- `bugs_cards/light_tickets.csv` - тестовый датасет в формате CSV со списком
  учебных багов.
- `sirius_tests/` - тесты, которые нужно запускать на этом этапе.
- `scripts/` - команды для настройки окружения и запуска учебных тестов.

Баги описаны в карточках внутри `bugs_cards/` в ветке `sirius-light-buggy`.
Для каждого бага карточка показывает исходный текст задачи, ручное
воспроизведение, ожидаемое поведение, команду запуска теста и файлы кода, с
которых удобно начать разбор.

Тесты для запуска лежат в `sirius_tests/`. На этапе погружения команда работает
именно с этими тестами, а не с полным test suite SymPy.

## Цель этапа

Познакомить команду с четырьмя вещами:

1. Что такое SymPy как пользовательская библиотека.
2. Как читать карточку бага: исходный текст, ручное воспроизведение,
   ожидаемое поведение, связанный тест и файл кода.
3. Какие учебные баги собраны в проекте, какое поведение считается ожидаемым (разобрать 2-3 бага по карточкам).
4. Перейти от ручного воспроизведения к тестам.

После этапа команда должна уметь:

- развернуть локальное окружение;
- переключаться между `sirius-light-buggy` и `sirius-light-golden`;
- запускать примеры использования SymPy;
- находить и читать карточки багов в `bugs_cards/`;
- запускать тесты;

## Структура fork

В репозитории есть две учебные ветки:

- `sirius-light-golden` - эталонная ветка с исправленным поведением.
- `sirius-light-buggy` - рабочая ветка с 10 внесенными регрессиями.

Обе ветки используют один и тот же тестовый файл:

```bash
sirius_tests/test_light_bugs.py
```

Полный test suite SymPy на этом этапе не нужен. Команда работает только с
небольшим набором учебных тестов.

## Карточки багов

В ветке `sirius-light-buggy` добавлена директория:

```bash
bugs_cards/
```

В ней лежат:

- `light_tickets.csv` - исходная таблица с 10 учебными багами.
- `template.md` - шаблон карточки бага.
- `01_...md` - `10_...md` - готовые карточки по всем багам.


Методически карточка должна быть первым документом по конкретному багу. После
неё команда переходит к запуску примера, затем к тесту, затем к чтению кода.

## Блок 1. Продуктовое погружение

Задача блока - показать, что баг в библиотеке виден не только в тестах, но и в
обычном пользовательском сценарии.

Рекомендуемый ход:

1. Открыть README и пройти раздел "Что такое SymPy".
2. Запустить пример с производной.
3. Запустить пример с матрицей.
4. Переключиться на `sirius-light-buggy`.
5. Открыть первую карточку:

```bash
bugs_cards/01_sympy__sympy-18621.md
```

6. Запустить ручное воспроизведение из карточки в `sirius-light-buggy`.
7. Затем запустить тот же пример в `sirius-light-golden`.
8. Зафиксировать разницу: в buggy-ветке пользователь получает ошибку, в
   golden-ветке получает корректный результат.

Второй пример удобно взять из карточки:

```bash
bugs_cards/09_sympy__sympy-17655.md
```

Он нужен для закрепления: команда видит другой тип ошибки, но тот же принцип
сравнения buggy/golden.

## Блок 2. Работа с bug-веткой

Задача блока - перейти от продуктового погружения к инженерной проверке:
запустить учебные тесты и сравнить поведение bug-ветки и golden-ветки.

Рекомендуемый ход:

1. Переключиться на `sirius-light-buggy`.
2. Выполнить `./scripts/setup_sirius.sh`.
3. Запустить все учебные тесты:

```bash
./scripts/run_sirius_tests.sh
```

4. Убедиться, что в `sirius-light-buggy` тесты падают. Это ожидаемое
   состояние: ветка содержит внесенные регрессии.
5. Переключиться на `sirius-light-golden`.
6. Выполнить `./scripts/setup_sirius.sh`.
7. Запустить все учебные тесты:

```bash
./scripts/run_sirius_tests.sh
```

8. Убедиться, что в `sirius-light-golden` все учебные тесты проходят.

## Ожидаемый результат этапа погружения

К концу этапа погружения команда не обязана исправить все баги. Достаточно,
если команда:

- понимает различие между `buggy` и `golden`;
- умеет запускать окружение и тесты;
- умеет открыть карточку бага и пройти её по шагам;
- умеет вручную воспроизвести 1-2 бага из карточек;
- понимает, что тест фиксирует ожидаемое поведение библиотеки;
- может назвать файл теста и примерный файл кода, который относится к багу;
- понимает, что `bugs_cards/light_tickets.csv` - источник списка задач, а
  markdown-карточки - удобный учебный маршрут по этим задачам.

## Что не нужно делать в этом блоке

- Не запускать полный test suite SymPy.
- Не начинать с архитектуры всей библиотеки.
- Не требовать сразу исправить все 10 багов.
- Не смешивать ручное воспроизведение и тесты: сначала поведение как пользователь,
  затем проверка как инженер.
- Не использовать golden-ветку как рабочую ветку для исправлений: она нужна для
  сравнения ожидаемого поведения.
